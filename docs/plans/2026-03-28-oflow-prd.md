# oflow — PRD, Technical Scope & Implementation Roadmap

**Date:** 2026-03-28
**Status:** Draft
**Version:** 0.1

---

## 1. Vision

oflow is a workflow automation layer that connects issue tracking boards to AI coding agents. It monitors a board for available tasks, hands them to a coding agent, and lets the agent execute a full structured development workflow autonomously — from codebase exploration through to an open PR.

**Core principle:** oflow is a guest in any repo. It provides tools and skills. It never owns the workflow logic — that belongs to the agent. It never forces its artifacts on host projects — `.oflow/` is the host's to keep or ignore.

---

## 2. Goals

- Reduce context-switching overhead for developers working from issue boards
- Make AI-assisted development auditable and reproducible via structured step artifacts
- Provide skills that are useful both inside the automated pipeline and standalone in Claude Code
- Be pluggable: different boards, different agents, same workflow skills
- Bootstrap itself: oflow's own development is managed by oflow

---

## 3. Non-Goals (v1)

- Parallel task execution (MVP: 1 task at a time)
- Conflict resolution between concurrent agent sessions
- Web UI or dashboard
- Self-healing / automatic retry on agent failure (log and stop)
- Support for GitLab, Jira, Azure DevOps (architecture supports them, implementation deferred)
- Opencode subagent orchestration parity with Claude Code (documented limitation)

---

## 4. User Stories

### Automated mode (daemon)
> As a developer, I run `oflow start` in my project. It monitors my GitHub board, picks up tasks labelled `oflow-ready`, spawns a Claude Code session for each, and opens a PR when done — without me touching anything.

### Manual mode (standalone skills)
> As a developer, I open Claude Code and invoke `/dev-workflow` with a task description. The full structured workflow runs in my current session using subagents for each step.

### Tool use inside agent
> As a Claude Code subagent running the "plan" step, I call `oflow state write plan` to persist my output and `oflow validate plan` to confirm the artifact meets the required schema before signalling completion.

### New workflow type
> As a team lead, I drop a `ci-fix-workflow.md` skill file into `.claude/skills/` and add a `ci-fix` workflow entry to `oflow.config.ts`. It immediately works with the existing daemon and board poller.

## 5. Core Features

### 5.1 Board Adapters
Pluggable adapters behind a `BoardAdapter` interface. MVP: GitHub Issues.

Each adapter implements:
- `listAvailableTasks(filter)` — returns tasks ready to pick up
- `claimTask(taskId)` — assign to bot, set status to in-progress
- `updateTaskStatus(taskId, status, message)`
- `getTaskContext(taskId)` — returns structured task context (title, description, labels, linked issues)

### 5.2 Agent Adapters
Pluggable adapters behind an `AgentAdapter` interface. MVP: Claude Code + Opencode.

Each adapter implements:
- `spawn(skill, context, options)` — start agent session with skill and task context
- `getStatus(sessionId)` — running | completed | failed
- `getLogs(sessionId)` — stream or read session output

### 5.3 oflow CLI Toolbox
Deterministic subcommands called by agent skills during workflow execution.

```
oflow board list              # list available tasks
oflow board pick              # claim next available task, print task context JSON
oflow board update <id>       # update task status/comment

oflow state init <task-id>    # create .oflow/runs/<task-id>/ directory
oflow state write <artifact>  # write artifact markdown to run directory
oflow state read <artifact>   # read artifact markdown from run directory
oflow state list              # list artifacts written so far

oflow validate <artifact>     # validate artifact against its schema
oflow pr create               # create PR, post summary comment on task
oflow run                     # start daemon (poll + spawn loop)
oflow status                  # show currently running sessions and slot usage
```

### 5.4 Workflow Skills
Markdown skill files usable standalone in Claude Code or invoked by the daemon.

**MVP workflows:**
- `dev-workflow` — full development cycle (explore → plan → implement → review → PR)

**Planned workflows:**
- `code-review-workflow` — review an open PR, post structured feedback
- `ci-fix-workflow` — pick up a failing CI run, diagnose and fix

Each workflow skill describes the full sequence of subagent steps and the transition criteria between them. Each step has a corresponding focused skill and produces a required artifact.

### 5.5 Step Skills (subagents)
Each step in a workflow is its own skill invoked as a Claude Code subagent:

| Skill | Input artifacts | Output artifact | Transition criterion |
|---|---|---|---|
| `exploration` | task context | `exploration.md` | All required sections present |
| `plan` | exploration.md | `plan.md` | Subtasks listed, approach defined |
| `review-plan` | plan.md | `plan-review.md` | Loop until verdict: PASS |
| `implement-subtask` | plan.md, impl-N-1.md | `impl-N.md` | Tests pass for subtask |
| `final-validation` | all impl-N.md | `validation.md` | Full test suite passes |
| `code-review` | all artifacts | `review.md` | No blockers found |
| `open-pr` | review.md | PR URL in task comment | PR opened, task updated |

### 5.6 State & Audit Trail
Every task run produces a directory at `.oflow/runs/<task-id>/`:

```
.oflow/
  runs/
    GH-42/
      task-context.json      # original task from board
      exploration.md
      plan.md
      plan-review.md         # last review iteration
      impl-1.md
      impl-2.md
      validation.md
      review.md
      run.log                # full agent session log
      summary.md             # posted to PR as comment
```

Host projects decide whether to commit `.oflow/` or gitignore it.
oflow's own repo gitignores `.oflow/`.

### 5.7 Configuration
Single `.env` file in the host project root (never committed):

```env
# Board
OFLOW_BOARD=github
OFLOW_GITHUB_TOKEN=ghp_...
OFLOW_GITHUB_REPO=owner/repo
OFLOW_GITHUB_PROJECT=My Board
OFLOW_TASK_LABEL=oflow-ready          # label that marks a task as pickable
OFLOW_TASK_IN_PROGRESS_LABEL=oflow-in-progress

# Agent
OFLOW_AGENT=claude-code               # claude-code | opencode
OFLOW_AGENT_MODEL=claude-opus-4-6     # passed to agent on spawn
OFLOW_MAX_CONCURRENT_TASKS=1

# Workflow
OFLOW_DEFAULT_WORKFLOW=dev-workflow

# Polling
OFLOW_POLL_INTERVAL_SECONDS=60
```

Optional `oflow.config.ts` for advanced configuration (custom workflow routing by label, custom artifact schemas).

### 6.2 Key Interfaces

```typescript
// Board adapter
interface BoardAdapter {
  listAvailableTasks(filter?: TaskFilter): Promise<Task[]>
  claimTask(taskId: string): Promise<Task>
  updateTaskStatus(taskId: string, update: TaskUpdate): Promise<void>
  getTaskContext(taskId: string): Promise<TaskContext>
}

// Agent adapter
interface AgentAdapter {
  spawn(options: SpawnOptions): Promise<Session>
  getStatus(sessionId: string): Promise<SessionStatus>
  getLogs(sessionId: string): Promise<string>
  waitForCompletion(sessionId: string): Promise<SessionResult>
}

// Task context passed to agent on spawn
interface TaskContext {
  id: string
  title: string
  description: string
  labels: string[]
  workflow: string           // which workflow skill to run
  repoPath: string           // absolute path to target repo
  runDir: string             // .oflow/runs/<task-id>/
}
```

### 6.3 Artifact Schema (example: plan.md)

All artifacts use YAML frontmatter for machine-readable fields, markdown body for human-readable content.

```markdown
---
artifact: plan
task_id: GH-42
status: draft | ready | approved
verdict: PASS | FAIL | PENDING
created_at: 2026-03-28T10:00:00Z
subtasks:
  - id: 1
    title: "Add BoardAdapter interface"
    estimate: small
  - id: 2
    title: "Implement GitHub adapter"
    estimate: medium
---

## Approach
...

## Subtasks
...

## Open Questions
...
```

Validated by Zod schemas in `src/state/schemas/`.

### 6.4 Daemon Loop (simplified)

```typescript
async function daemonLoop(config: Config) {
  while (true) {
    const freeSlots = config.maxConcurrent - activeSessions.size
    if (freeSlots > 0) {
      const tasks = await board.listAvailableTasks()
      if (tasks.length > 0) {
        const task = await board.claimTask(tasks[0].id)
        const session = await agent.spawn({ skill: task.workflow, context: task })
        activeSessions.set(task.id, session)
        log.info(`Started session for task ${task.id}`)
      }
    }
    // check completions
    for (const [taskId, session] of activeSessions) {
      const status = await agent.getStatus(session.id)
      if (status !== 'running') {
        await handleCompletion(taskId, session, status)
        activeSessions.delete(taskId)
      }
    }
    await sleep(config.pollIntervalMs)
  }
}
```

## 7. Artifact Flow Between Steps

```
task-context.json
      │
      ▼
[exploration subagent] ──────▶ exploration.md
      │
      ▼
[plan subagent] ──────────────▶ plan.md
      │
      ▼ (loop)
[review-plan subagent] ────────▶ plan-review.md
   verdict==PASS?
      │ yes
      ▼
[implement-subtask × N] ───────▶ impl-1.md ... impl-N.md
  (sequential, tests after each)
      │
      ▼
[final-validation subagent] ───▶ validation.md
      │
      ▼
[code-review subagent] ────────▶ review.md
      │
      ▼
[open-pr subagent] ────────────▶ PR opened, task updated
```

Each subagent reads all previously written artifacts from `.oflow/runs/<task-id>/` for full context. This is the contract: **write your artifact, read what came before.**

## 8. Implementation Roadmap

### Phase 0 — Scaffold (Week 1)
Get a working TypeScript project with CI.

- [ ] Init TypeScript project (`tsconfig`, `package.json`, `eslint`, `vitest`)
- [ ] Set up GitHub Actions: lint + test on PR
- [ ] Implement `config/loader.ts` — loads `.env`, validates required fields
- [ ] Create `.env.example`
- [ ] Create `.gitignore` with `.oflow/` entry
- [ ] `oflow --version` and `oflow --help` work

### Phase 1 — Board Adapter (Week 1-2)
oflow can talk to GitHub Issues.

- [ ] Define `BoardAdapter` interface and `Task` / `TaskContext` types
- [ ] Implement `GitHubBoardAdapter` using GitHub REST API
  - list issues with label filter
  - claim task (assign + add label)
  - update status (comment + label swap)
  - get task context
- [ ] `oflow board list` — prints available tasks
- [ ] `oflow board pick` — claims task, writes `task-context.json` to `.oflow/runs/<id>/`
- [ ] `oflow board update <id> --status <status> --message <msg>`
- [ ] Unit tests with mocked GitHub API

### Phase 2 — State & Validation (Week 2)
oflow can read/write artifacts and validate schemas.

- [ ] Implement `StateManager` — CRUD for `.oflow/runs/<task-id>/`
- [ ] Define Zod schemas for all artifact types
- [ ] `oflow state init <task-id>`
- [ ] `oflow state write <artifact> <file>`
- [ ] `oflow state read <artifact>`
- [ ] `oflow validate <artifact>` — exits 0 on pass, 1 on fail with field errors
- [ ] Unit tests for all schemas

### Phase 3 — Skills (Week 2-3)
The full dev-workflow is usable standalone in Claude Code.

- [ ] Write `skills/steps/exploration.md` — codebase exploration subagent skill
- [ ] Write `skills/steps/plan.md` — planning subagent skill
- [ ] Write `skills/steps/review-plan.md` — plan review subagent with PASS/FAIL verdict
- [ ] Write `skills/steps/implement-subtask.md` — single subtask implementation skill
- [ ] Write `skills/steps/final-validation.md` — full test suite validation skill
- [ ] Write `skills/steps/code-review.md` — code review subagent skill
- [ ] Write `skills/steps/open-pr.md` — PR creation and board update skill
- [ ] Write `skills/dev-workflow.md` — top-level skill that orchestrates all steps via Agent tool
- [ ] Manual end-to-end test: run `/dev-workflow` in Claude Code on a real GitHub issue

### Phase 4 — Agent Adapters (Week 3)
oflow can spawn and monitor coding agent sessions.

- [ ] Define `AgentAdapter` interface and `Session` / `SessionResult` types
- [ ] Implement `ClaudeCodeAdapter`
  - spawn: `claude --skill dev-workflow` with context injected via stdin/env
  - monitor: watch process exit code
  - logs: capture stdout/stderr to `run.log`
- [ ] Implement `OpencodeAdapter` (best-effort, single-context mode — no native subagents)
- [ ] `oflow status` — shows active sessions

### Phase 5 — Daemon (Week 4)
oflow monitors the board and spawns agents automatically.

- [ ] Implement `poller.ts` — polling loop with configurable interval
- [ ] Implement `scheduler.ts` — slot management (MVP: 1 slot)
- [ ] Implement `spawner.ts` — ties poller + scheduler + agent adapter together
- [ ] `oflow run` — starts the daemon, logs to stdout
- [ ] Graceful shutdown on SIGTERM/SIGINT (wait for active session to complete)
- [ ] End-to-end test: label a GitHub issue `oflow-ready`, run `oflow run`, watch PR open

### Phase 6 — oflow builds itself (Week 4-5)
Use oflow to develop oflow.

- [ ] Create first real GitHub issues in this repo for remaining work
- [ ] Label them `oflow-ready`
- [ ] Run `oflow run` on the oflow repo
- [ ] Fix whatever breaks
- [ ] This phase is the real integration test

### Phase 7 — Hardening (Week 5-6)
Make it reliable enough to run unattended.

- [ ] `oflow pr create` — standalone PR creation tool
- [ ] Session timeout handling (kill + mark failed after N minutes)
- [ ] Retry logic for transient board API failures
- [ ] `summary.md` auto-posted as PR comment
- [ ] `oflow install` — installs as launchd/systemd service

## 9. Future Workflows

The architecture is workflow-agnostic. Adding a new workflow is:
1. Write the skill file in `skills/`
2. Optionally add step skills in `skills/steps/`
3. Route it via label in `oflow.config.ts`

**Planned workflows:**
- `code-review-workflow` — triggered by PR-ready label, posts review comment
- `ci-fix-workflow` — triggered by failing CI, diagnoses and fixes
- `triage-workflow` — labels and estimates incoming issues
- `docs-workflow` — generates or updates documentation from code changes

## 10. Open Questions

1. **Opencode subagents** — does opencode support spawning subagents natively? If not, `dev-workflow` skill needs an opencode-compatible variant that runs all steps in one context.
2. **Skill installation** — how does the host project get the oflow skills into Claude Code's skill path? Options: `oflow install-skills` copies them, or we publish as an npm package that includes a post-install hook.
3. **PR creation tool** — use `gh` CLI (requires `gh` installed) or GitHub REST API directly? Prefer `gh` for MVP.
4. **Agent context injection** — how does the agent receive `task-context.json` on spawn? Options: env var with JSON, stdin, or a known file path. Likely `--context-file` arg.
